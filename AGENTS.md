- ⚠️ Expo a changé : lire https://docs.expo.dev/versions/v56.0.0/ avant d'écrire du code.
- Lancer : `npm run web` (port 8081) ou `npm run ios`.
- Vérif visuelle : NE PAS utiliser 8081 (occupé par le `npm run web` de l'utilisateur). Lancer un preview sur 8090.
- **Déployé en ligne** : web sur GitHub Pages → https://brgkevin-arch.github.io/Kyroz-app/ (dépôt public `brgkevin-arch/Kyroz-app`, auto-deploy `deploy.yml` à chaque push sur `main`). ⚠️ `baseUrl` doit rester dans `app.json > expo.experiments.baseUrl = "/Kyroz-app"` (sinon page blanche). Le fondateur publie via GitHub Desktop (Commit → Push), pas le terminal. Ne jamais « Re-run all jobs » sur un vieux run (redeploie une version périmée) ; pour forcer : Actions → Run workflow.

## Décisions de fond (verrouillées — cf. CLAUDE.md)
- **Freemium large** : core loop 100% gratuit, sans clé API. Monétisation = features avancées (à définir).
- **Moteur de génération LOCAL** (`lib/planEngine.ts`) = principal. API Claude (`lib/generatePlan.ts`) seulement si `EXPO_PUBLIC_ANTHROPIC_API_KEY` (sk-ant-…) définie, sinon fallback local auto.
- **Supabase** = auth + **sync des données câblée** (offline-first, miroir par user, RLS stricte — cf. `lib/sync.ts`). Exceptions volontaires : plan non synchronisé (déterministe) ; photos de progression LOCAL-ONLY (RGPD).

## FAIT
- **Onboarding 8 étapes** (`app/(auth)/onboarding.tsx`) : sexe/âge/poids/taille **+ masse grasse optionnelle**, activité, 6 objectifs, macros auto/manuel, préférences alimentaires, variété, jours + repas à la carte + emphase, récap. Auto-génère le plan à l'arrivée.
- **TDEE/macros** (`lib/tdee.ts`) : **Katch-McArdle si % de masse grasse connu** (BMR sur masse maigre, protéines calées sur la masse maigre), sinon Mifflin par sexe. 6 objectifs, garde-fous (min kcal, âge ≥16, %MG borné 3–60).
- **Modes macros** : `auto` (tout calculé) · **`percent`** (option B, contrôle total : kcal suit le poids ; **protéines réglables en g/kg** via `protein_per_kg` avec repère conseillé par objectif `recommendedProteinPerKg` ; glucides/lipides répartis selon `carb_ratio` = % glucides de l'énergie non-protéique → `macrosPercent`). `manual` = legacy (grammes figés, plus proposé en UI). UI `components/MacroSplit.tsx` (steppers **±1 %** glucides et **±0,1 g/kg** protéines + saisie directe + **aperçu live des grammes** + note conseillée), onboarding (étape 4) + profil (MacroEditor). `carb_ratio`/`protein_per_kg` synchronisés (colonnes + migration `schema.sql`).
- **Précision masse grasse** : `body_fat_pct` (`lib/types.ts`) — **désormais REQUIS à l'onboarding** (étape 1 ne valide pas sans silhouette/`%` choisi), car indispensable à un plan bien adapté ; reste éditable (optionnel) dans le profil. `components/BodyFatPicker.tsx` = sélecteur 6 niveaux + saisie manuelle, avec **rendus 3D détourés** (`assets/bodyfat/{male,female}-1..6.png`, une image par niveau et par sexe, fond transparent → s'intègre au thème clair/sombre). Sources archivées dans `assets/bodyfat/_source/`. (L'ancien `components/BodySilhouette.tsx` — silhouettes SVG — a été supprimé.) Câblé onboarding (étape 1) + profil (InfoEditor). - **Édition de recette (overrides perso)** : registre dans `lib/recipes.ts` (`getEffectiveRecipes`/`getRecipeById`/`getBaseRecipe`/`setRecipeOverrides`) = base ⊕ perso, lu par le moteur, le garde-manger et la génération (plus de `RECIPES_PLACEHOLDER` direct). - **Check-in « ton plan te convient ? »** (`hooks/usePlanCheckin.ts`, `components/PlanCheckin.tsx`) : proposition périodique (tous les 14 j, jamais le J1) sur l'écran Plan → sheet d'ajustements en 1 tap (variété max, moins de prépa, nouvel objectif→Profil, régénérer).
